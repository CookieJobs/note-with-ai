## 核心目标
- 保存后笔记立即出现在【笔记信息流】，不等待富化结果
- 标题、标签（关键词）、相关笔记检索走异步流程，并在各自位置显示加载占位
- 整体体验：输入 → 保存 → 立刻看到卡片；其上元素逐步填充

## 现状梳理
- 前端已在保存后插入返回的笔记并异步调用向量与相关笔记：`frontend/src/app/notes/page.tsx:710-799`；相关笔记区已有“查找中...”加载态：`frontend/src/app/notes/page.tsx:900-930`
- 标题显示位置：`frontend/src/app/notes/page.tsx:329-339`；关键词显示位置：`frontend/src/app/notes/page.tsx:417-449`
- 后端创建接口在保存前调用摘要生成（同步）：`backend/routes/notes.ts:24-46`
- 后端 embedding 异步接口：`backend/routes/notes.ts:64-78`
- 后端局部更新并支持 `autoSummarize`：`backend/routes/notes.ts:124-184`

## 方案概述
- 将“标题与关键词生成”从创建接口的同步步骤改为“保存后异步触发”，使创建接口快速返回
- 前端采用“乐观插入临时笔记 + 分部加载态”，在各子区域独立显示骨架占位并逐步替换为真实数据
- 复用现有 `/:id/embed` 与 `PATCH /:id` 的 `autoSummarize` 来异步生成向量、标题与关键词

## 前端改动
- 乐观插入临时笔记
  - 保存按钮点击后，立即在列表顶部插入临时笔记对象：`_id: 'temp-'+uuid`、`content: 当前文本`、`title: ''`（或首行截断作为临时标题）、`keywords: []`、`createdAt: now`
  - 为临时笔记添加前端状态字段：`enriching: true`
- 异步触发三条任务（互不阻塞）
  - 创建真实笔记：`POST /api/notes`；返回后用真实 `_id` 替换临时项
  - 生成向量：`POST /api/notes/:id/embed`（已存在，保持不变）
  - 生成标题与关键词：`PATCH /api/notes/:id`，payload 含 `content: 原文` 和 `autoSummarize: true`（即使内容未变也会触发）
- 占位与加载态
  - 标题：当 `note.title` 为空且 `enriching===true`，在 `ModernNoteCard` 标题区显示骨架占位（如“标题生成中…”）
  - 关键词：当 `note.keywords.length===0` 且 `enriching===true`，显示骨架标签占位（3~5 个 shimmer chip）
  - 相关笔记：保持现状，已在保存后立即设置 `relatedLoading=true` 并显示“查找中...”：`frontend/src/app/notes/page.tsx:905-906`
- 状态落地
  - 创建返回后：用真实笔记替换临时项，继续保持 `enriching=true`
  - 摘要返回后：更新 `title/keywords` 并置 `enriching=false`
- 并发与回退
  - 若摘要失败：保留临时标题（首行或空），允许用户手动编辑；在关键词区显示“生成失败”轻提示（不打断使用）
  - 保持现有正文/关键词编辑的乐观并发与冲突处理：`frontend/src/app/notes/page.tsx:76-115, 125-164`
- 涉及文件
  - `frontend/src/app/notes/page.tsx`：新增临时笔记插入、`enriching` 状态、摘要触发与骨架占位渲染
  - `frontend/src/app/notes/notes.module.scss`：新增骨架样式（标题、关键词 chip shimmer）
  - 如需复用骨架，可抽出通用组件至 `frontend/src/components/`（可选）

## 后端改动
- 创建接口快速返回
  - 将 `POST /api/notes` 改为仅保存原文与一个“临时标题”（如首行截断），`keywords: []`，立即返回 201
  - 把现有同步 `summarizeNote(content)` 从创建接口移除：`backend/routes/notes.ts:24-46`
- 异步富化
  - 保留 `POST /api/notes/:id/embed`，由前端在创建成功后调用
  - 复用 `PATCH /api/notes/:id` 的 `autoSummarize`：前端发送 `{ content, autoSummarize: true }` 触发服务端生成并落库
  - 若希望接口更语义化，可新增 `POST /api/notes/:id/summarize`（内部调用同样逻辑），但首版可不增接口
- 不改动数据模型
  - 继续使用现有 `Note` 字段；前端 `enriching` 为本地 UI 状态，不入库

## 验证与度量
- 交互验证
  - 保存后 <200ms 卡片出现在信息流顶部；标题/关键词区域显示骨架，占位后逐步填充
  - 相关笔记区出现“查找中...”，随后展示列表或“未找到相关笔记”
- 接口验证
  - 创建接口 95% 响应 <300ms；`/:id/embed` 与 `PATCH /:id autoSummarize` 成功率与耗时监控
- 回归用例
  - 标题编辑（`POST /api/notes/:id`）、正文编辑与冲突（409）、关键词编辑流程
