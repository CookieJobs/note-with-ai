## 目标
- 在整个仓库建立一致的文档与代码注释规范，便于维护与协作。
- 明确更新责任：任何功能、架构、写法的改动，完成工作后必须同步更新对应目录文档与文件头注释。
- 通过自动化脚本与提交前校验，减少遗漏。

## 范围
- 根目录与各子目录的 README 文档规范。
- 所有代码类文件的文件头三行极简注释（Input / Output / Pos），并附更新提示语。
- 忽略类型：`*.md`、`*.json`、`package-lock.json`、`yarn.lock`、`*.ico`、`*.png`、`*.jpg`、`*.svg` 等静态与数据文件。

## 标准与模板
### 根目录 README 规范段落（新增）
- 要求：强调“任何功能、架构、写法更新必须在工作结束后更新相关目录的子文档与文件头注释”。
- 示例段落：
  - 贡献约定：每次变更完成后，务必更新受影响的目录 README 与相关文件的头部注释（Input/Output/Pos）。未按约定的提交将在校验环节被拒绝。

### 目录级 README 模板（每个目录一个）
- 结构（≤3 行架构说明 + 文件清单 + 更新声明）：
  1. 本目录职责：用 1–3 行说明该目录的定位与职责。
  2. 文件清单：逐项列出文件“名字 / 地位 / 功能”（地位示例：组件、路由、服务、模型、工具、样式、脚本）。
  3. 更新声明：一旦我所属的文件夹有所变化，请更新我。
- 示例：
  - 架构说明：本目录用于后端路由注册与请求分发。与控制器、鉴权中间件协作。
  - 文件清单：
    - `notes.ts` — 路由 — 笔记相关的 REST 路由入口
    - `auth.ts` — 路由 — 用户登录注册与令牌刷新入口
  - 声明：一旦我所属的文件夹有所变化，请更新我。

### 文件头三行极简注释模板
- TS/TSX/JS：
  ```
  /*
  Input: 依赖的外部模块与接口（示例：service/search、模型 Note）
  Output: 对外暴露的能力（示例：默认导出组件、导出函数 fetchNotes）
  Pos: 在系统中的地位（示例：前端页面路由、后端控制器、通用工具）
  Note: 一旦我被更新，务必更新我的开头注释，以及所属的文件夹的 README
  */
  ```
- CSS/SCSS：
  ```
  /*
  Input: 依赖的样式变量/混入（示例：globals.css）
  Output: 提供的样式作用域（示例：NoteCard 组件样式）
  Pos: 样式层级角色（示例：组件级样式/全局样式）
  Note: 一旦我被更新，务必更新我的开头注释，以及所属的文件夹的 README
  */
  ```
- Shell/脚本文件（`*.sh`/`*.js`工具脚本）：
  ```
  # Input: 外部命令或环境变量（示例：node、npm、.env）
  # Output: 执行结果/生成的文件（示例：构建产物、修复报告）
  # Pos: 辅助脚本（示例：开发/部署/修复脚本）
  # Note: 一旦我被更新，务必更新我的开头注释，以及所属的文件夹的 README
  ```

## 自动化与校验
- 新增工具脚本（位于 `scripts/doc-tools/`）：
  - `doc:scan`：扫描整个仓库，输出缺失目录 README 与缺失/不合规文件头的清单。
  - `doc:inject`：为缺失的文件头插入标准模板（以“待补充”占位），不覆盖已有头注释。
  - `doc:check`：校验规则，若存在缺失或模板占位未被人工完善的文件，进程退出码为 1。
- 提交前钩子：引入 `husky`，在 `pre-commit` 执行 `doc:check`，阻止不合规提交。
- Lint 规则（可选加强）：
  - 前端：自定义 ESLint 规则或插件，检测文件首注释块是否包含 `Input/Output/Pos/Note` 关键字。
  - 样式：Stylelint 自定义规则（或简单脚本），检测 CSS/SCSS 顶部注释块。

## 首批落地目录（将创建/更新 README）
- `backend/`，以及：`backend/config`、`backend/controllers`、`backend/middleware`、`backend/models`、`backend/routes`、`backend/scripts`、`backend/services`、`backend/utils`。
- `frontend/`，以及：`frontend/src/app`、`frontend/src/app/auth`、`frontend/src/app/chat`、`frontend/src/app/login`、`frontend/src/app/notes`、`frontend/src/app/register`、`frontend/src/components`（含 `icons`）、`frontend/src/hooks`、`frontend/src/lib`、`frontend/src/styles`、`frontend/src/utils`。
- 根级：`scripts/`（已有 README，将补充规范段落），其他必要目录按扫描结果补齐。

## 执行步骤
1. 更新根 `README.md`，加入“更新责任”规范段落。
2. 在上述目录放置（或更新）极简 `README.md`，包含 3 行架构说明 + 文件清单 + 更新声明。
3. 为所有代码文件插入文件头模板（仅在缺失时；已有则保留）。
4. 新增 `scripts/doc-tools`：实现 `scan/inject/check` 三脚本。
5. 安装并配置 `husky` 的 `pre-commit`，运行 `doc:check`。
6. 首次执行 `doc:inject` 后，人工完善关键文件的 `Input/Output/Pos`（特别是路由、服务、页面组件）。

## 验证
- 本地运行 `npm run doc:scan`：确认无遗漏目录与文件。
- 执行 `npm run doc:check`：若返回非 0，按输出补齐或完善注释。
- 修改示例文件（如 `frontend/src/app/notes/page.tsx`）后再次检查，确保流程有效。

## 维护要求
- 每次合并前：保证受影响目录 README 与相关文件头已更新。
- 每次大改动（移动目录/文件）：先更新目录 README，再更新文件头注释，最后通过 `doc:check` 校验。

## 风险与注意
- 注释插入不应影响编译：模板使用对应语言的合法注释格式。
- 避免覆盖已有专业说明：`doc:inject` 仅在文件顶部缺少规范块时插入占位。
- JSON/静态资源不加注释：由扫描器忽略。
