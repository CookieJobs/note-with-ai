## 原因分析
- React 18 开发模式下默认启用 StrictMode，useEffect 会在 mount→unmount→mount 的双阶段检查中执行两次；因此 `CareAssistantPanel` 的 `useEffect(() => fetchIntro(), [])` 会调用 2 次。
- 新建会话保存后，当前会话 ID 可能从本地 `local_...` 切换为服务端 ID，若组件被重新挂载或使用了 `key` 绑定 ID，会触发二次拉取。

## 修复策略
- 在 `CareAssistantPanel.tsx` 内部增加一次性防重触发：
  - 使用 `useRef<boolean>(false)` 标记是否已拉取，`useEffect` 中先判断再调用。
  - 为 `fetchIntro` 增加 `AbortController`，避免卸载后仍更新状态导致闪烁。
- 仅在“新建且空会话”渲染面板（已实现）；避免因会话 ID 切换造成的重复 mount 不必要的重复拉取。

## 代码改动
- 文件：`frontend/src/components/CareAssistantPanel.tsx`
  - 增加：
    - `const fetchedRef = useRef(false)`
    - `useEffect(() => { if (fetchedRef.current) return; fetchedRef.current = true; fetchIntro(); }, [])`
    - 在 `fetchIntro` 内部使用 `AbortController`：组件卸载时 `controller.abort()`；请求失败/被中止时静默
- 验证：
  - 开发模式新建会话仅首屏拉取一次；点击“换一个”只触发一次新的拉取；生产模式行为一致。

## 备选（如仍有二次触发）
- 将面板与会话 ID 解绑（不设置 `key`），确保切换到服务端 ID 时不会 remount。
- 如需严格“一次每会话”，可以从 ChatPage 将 `currentSession.id` 作为依赖传入面板，并在面板内部用 `lastSessionIdRef` 比对，仅当 ID 不同且消息仍为空时才允许首拉取。
